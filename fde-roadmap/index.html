Forward Deployed Engineer
Learning Roadmap

A focused guide on what to learn (and where) to be ready for a Forward Deployed Engineer / Applied AI Engineer role at any frontier AI lab or enterprise AI company. Organized by topic, ordered roughly from foundations → advanced → role-shaped soft skills.

Given a strong SWE background, the biggest deltas are usually: LLM-native engineering patterns, agent infrastructure (MCP, skills, sub-agents), evaluation discipline, and the consulting half of the job — discovery, scoping under ambiguity, and translating between executives and engineers.

1. Foundations of LLM-native engineering

The mental shift from "API that returns text" to "stochastic component you have to engineer around." Read broadly across labs — not just one.

  • Anthropic - Building Effective AI Agents: the canonical essay distinguishing workflows (predefined code paths) from agents (LLMs dynamically directing tool use). Covers prompt chaining, routing, parallelization, orchestrator-workers, evaluator-optimizer. Read essay →
  • Anthropic - Effective Context Engineering for AI Agents: the follow-up, defining "context engineering" as the next phase after prompt engineering. Read essay →
  • OpenAI - A Practical Guide to Building Agents (PDF): Download guide →
  • Chip Huyen - AI Engineering (book): the most comprehensive practitioner text. Covers eval, RAG, fine-tuning tradeoffs, inference optimization end-to-end.
  • Simon Willison's blog: pragmatic, current, vendor-agnostic. Visit blog →
  • Hamel Husain's blog, especially the eval posts: Visit blog →
  • Eugene Yan's blog: applied ML and LLM patterns from a practitioner who's shipped at scale. Visit blog →

2. The major API surfaces

You should be fluent in at least one frontier API end-to-end, and literate in the others — customers always ask "why not the other guy."

  • Anthropic Claude docs: docs.claude.com
  • OpenAI API docs: platform.openai.com/docs
  • Google Gemini API docs: ai.google.dev/gemini-api/docs
  • Meta Llama (open weights): llama.com

Core concepts to internalize regardless of provider:

  • The chat completions / messages loop and streaming
  • Tool use / function calling — schema design, the tool_call → tool_result loop, parallel tool calls
  • Structured outputs (JSON mode, schemas)
  • Prompt caching (huge cost/latency lever in production)
  • Batch APIs (often 50% cost for non-realtime workloads)
  • Reasoning / extended-thinking models and when they're worth the latency
  • Context windows, attention budgets, and how performance degrades at the tail

Pick a deliberate sub-project: ship one real app on the API of your target lab, instrumented with caching + tool use + a proper eval harness. Forces depth.

3. Prompt engineering and context engineering

The literature is much better than it was a year ago.

  • Anthropic prompt engineering guide: View guide →
  • OpenAI prompting guide: View guide →
  • DAIR.AI Prompt Engineering Guide (vendor-agnostic): promptingguide.ai

Patterns worth being fluent in: few-shot, chain-of-thought, XML/structured tagging, prefilling, self-consistency, ReAct, reflexion, plan-and-execute.

4. Agent infrastructure and protocols

This is where most enterprise FDE work lives today. The patterns are converging across labs.

  • Model Context Protocol (MCP) — open standard for connecting LLMs to tools and data. Originated at Anthropic but adopted broadly. modelcontextprotocol.io and GitHub →
  • Agent Skills — open standard for packaging procedural knowledge (a SKILL.md + scripts) that an agent loads on demand. agentskills.io
  • OpenAI Agents SDK: Documentation →
  • LangGraph (LangChain's graph-based orchestrator): Documentation →
  • CrewAI, AutoGen, Mastra: know they exist, know the tradeoffs. Most senior engineers in this space prefer thinner abstractions over heavyweight frameworks.

Concrete build to do once: ship one MCP server publicly. Pick something an enterprise would actually care about (e.g. a wrapper around an internal API shape — Jira-like, ServiceNow-like, Salesforce-like). Two transports (stdio + HTTP), one weekend.

5. Retrieval and RAG (without over-indexing)

RAG is no longer the centerpiece it was in 2023, but you need to know it cold because most enterprise deployments still involve it.

  • Anthropic - Contextual Retrieval: Read more →
  • Pinecone Learning Center: pinecone.io/learn
  • LlamaIndex docs: docs.llamaindex.ai
  • Eugene Yan - "Patterns for Building LLM-based Systems & Products": Read article →

Things to actually understand: chunking strategies, hybrid search (BM25 + dense), reranking, evaluation (recall@k, MRR), when to use a graph/structured approach instead. Don't spend weeks building a vector DB from scratch — understand at the level of "when to use it, how to evaluate it."

6. Evaluation frameworks (the part most engineers underweight)

FDEs ship things to customers who need to justify spend. No evals = no production deployment.

  • Hamel Husain - "Your AI Product Needs Evals": Required reading →
  • Anthropic eval docs: Documentation →
  • OpenAI Evals (framework + cookbook): GitHub →
  • Inspect AI (UK AISI's open-source framework, widely used in enterprise): inspect.ai-safety-institute.org.uk
  • Braintrust: braintrust.dev
  • Langfuse (open-source): langfuse.com
  • Promptfoo: promptfoo.dev

Patterns to learn: LLM-as-judge (and its failure modes), pairwise comparison, rubric-based grading, golden datasets, regression suites, online eval via user signals.

Pick one framework and ship an eval harness with it. The framework matters less than having done it once seriously.

7. Observability, security, and safety

The boring parts that determine whether a deployment actually survives in production.

Observability

  • Langfuse, Braintrust, Arize Phoenix: phoenix.arize.com
  • Helicone: helicone.ai
  • Patterns: trace every LLM call, log prompts + completions + tokens + latency + cost, sample for offline eval.

8. Enterprise vertical context

The JD almost always calls out "financial services, healthcare/life sciences, or another enterprise vertical" as a plus. Pick one and go deep enough to hold a 30-minute discovery conversation with a domain expert.

  • Financial services: SR 11-7 (model risk management), trade lifecycle basics, KYC/AML workflows, how compliance reviews AI systems. Reference customers: Bridgewater, Brex, JPMorgan's AI work.
  • Healthcare / life sciences: HIPAA, PHI handling, FDA's framework for AI/ML-based SaMD, clinical documentation workflows, EHR integration patterns.
  • Legal: contract review workflows, document retention, privilege. Reference companies: Harvey, Hebbia, Ironclad.
  • Customer service / CX: ticketing, deflection, agent assist, voice. Reference companies: Intercom (Fin), Sierra, Decagon.

Each frontier lab publishes customer stories — read 5-10 across verticals to see what real deployments look like.

9. The forward-deployed motion (the consulting half)

This is what most engineers underweight, and it's often what separates strong FDE candidates from strong SWE candidates.

  • Palantir FDE writing — Palantir invented this role. Read their playbook: Day in the life → and search YouTube for "Palantir forward deployed" talks.
  • The Mom Test by Rob Fitzpatrick — short book, the customer-discovery bible.
  • SPIN Selling by Neil Rackham — the question structure FDEs use in technical scoping calls.
  • The Pyramid Principle by Barbara Minto — writing/communicating for executives.
  • An Elegant Puzzle by Will Larson, and his blog lethain.com — working under ambiguity at staff level.
  • High Output Management by Andy Grove — still the best book on managing throughput across functions.

A practical exercise: take one of your past complex projects and write it up as a case study — situation, customer/stakeholder goal, constraints, what you built, measured outcome. FDE interview loops lean heavily on these.

10. Building your evidence portfolio (what to ship before applying)

Concrete artifacts an interviewer can read in 10 minutes carry more weight than anything else. Aim for 3-4 of these:

  1. An MCP server for a non-trivial enterprise-shaped API. Open-source on GitHub with a clear README and a 60-second demo video.
  2. A custom Agent Skill (or equivalent for whichever lab you're targeting) that does something you actually use.
  3. A small agent that solves a real workflow you do every week. Show eval results, not just the demo.
  4. A teardown post: pick one of the frontier engineering posts above, implement the pattern, write 800-1200 words on what worked and what didn't. This is exactly the kind of artifact FDEs produce internally.
  5. A "deployment pattern" writeup: pick a customer profile (e.g. "mid-size insurance company doing claims triage"), sketch the architecture (which cloud, RAG approach, eval setup, guardrails). One page.

11. Suggested 8-week sequence

WEEK 1
Rebuild one piece of an existing app of yours on a frontier API with tool use + prompt caching.
WEEK 2
MCP. Ship one MCP server publicly.
WEEK 3
Agent Skills (or your target lab's equivalent). Read 3 reference skills. Ship one custom skill that composes with your MCP server.
WEEK 4
Evals. Pick one framework. Add an eval harness to the Week 1 project. Get a baseline number.
WEEK 5
Multi-agent + sub-agents. Build one orchestrator + 2 sub-agents pattern.
WEEK 6
Deployment surfaces. Deploy your agent to Bedrock or Vertex. Document the security model and data flow.
WEEK 7
Vertical depth. Pick one vertical from §9. Build a domain-shaped demo on top of your existing infrastructure.
WEEK 8
Polish. Write up 2-3 artifacts as case studies. Practice talking through them out loud — discovery questions, tradeoffs, what you'd do differently.

12. Things explicitly worth not over-investing in

  • Building a vector DB from scratch. The job is about deploying LLMs well, not winning a RAG benchmark.
  • Fine-tuning. Most frontier labs' top models aren't user-fine-tunable, and the FDE motion almost never needs it. Knowing why prompting + context engineering replace most fine-tuning use cases is more valuable than fine-tuning skill itself.
  • Framework depth for its own sake. Know LangChain / LlamaIndex / CrewAI exist; customers use them. Don't build your reference implementations on top of heavyweight frameworks — most senior engineers in this space prefer thin, direct API calls.
  • Chasing every new model release. Frontier moves weekly. Build for capability classes (reasoning / tool-using / cheap-and-fast / multimodal), not specific model names.
